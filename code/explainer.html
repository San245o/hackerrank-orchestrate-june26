Multi-Modal Evidence Review — Solution Explainer

For each damage claim (car / laptop / package) the system reads the claim conversation, the submitted image(s), the user's history row, and the minimum evidence requirements, then produces the required 14-column row: evidence sufficiency, risk flags, issue type, object part, the supported/contradicted/not-enough-info decision, supporting image IDs, image validity, and severity. Images are the source of truth; history only adds risk context.

Two strategies were built and benchmarked on the labeled sample, then the winner produced output.csv for the 44-row test set.

Why exact-match looks ~25% (and why that is mostly two soft fields)

Exact match requires all seven fields to be right simultaneously. The decision fields are strong (claim_status 85%, evidence/valid_image 95%), but two inherently subjective sub-labels cap it:

  • severity ~50% exact but 90% within one level — misses are almost all off-by-one (e.g. medium vs high), a near-miss, not a flip.
  • issue_type ~45% — confusions are between adjacent categories (crack ↔ glass_shatter, scratch ↔ dent, stain ↔ water_damage).

So 0.95 × 0.95 × 0.85 × … × ~0.5 × ~0.45 ≈ 0.25. The pipeline gets the actual decision right ~85% of the time; exact-match is held down by two subjective fields. Note the sample is only 20 rows (each row = 5 points) and the model is not perfectly deterministic, so field-level numbers carry ±5–10 pt run-to-run noise — the refinements below moved the right direction but are within that band.

What the refinement changed (row-wise)

Diagnosing every per-row mismatch surfaced three systematic, fixable errors, addressed without hardcoding any answer:

  • risk_flags +10pt — the system missed damage_not_visible on contradicted "no damage" claims (added a rule), and over-added user_history_risk from numeric claim-count thresholds (now only the explicit history_flags marker triggers it).
  • severity — kept conservative anchors (do not inflate), recovering to 50% / 90%-within-1.
  • decision protected — an aggressive "contradict ⇒ none" instruction was tested and reverted because it destabilized claim_status (90%→65% in one run); the final prompt keeps the decision rubric untouched, holding claim_status at ~85%.

4. Why per-row was chosen for output.csv

Chosen: per-row (gemini-3.1-flash-lite).
  • Accuracy: ties or beats batched on every field (exact-match 20% vs 15%); the 3-stage pipeline buys no real accuracy here.
  • Cost: cheaper — batched's strong models emit several times more output tokens.
  • Reliability: per-row's small flash-lite calls ran clean every time; batched depends on the capacity-constrained gemini-3-flash-preview (~20 req/min, transient 503s on free tier).
  • When batched wins: only if the daily request cap (RPD) — not tokens/cost — is the hard limit, since it makes ~3.1× fewer requests.

6. Engineering: cost, latency, rate limits

  • Thinking off — thinkingBudget=0. These Gemini 3.x models otherwise spend the output budget "thinking", which truncated long JSON and ~10×'d output cost. Disabling it cut latency and cost with no accuracy loss observed.
  • Caching — every response cached by prompt + model (+ image bytes) under code/.cache/; reruns and the evaluation never repeat an identical call.
  • Throttling & retry — 5 s spacing; on HTTP 429 the client honors the server's retryDelay instead of blind retries (blind retries also burn the per-minute quota).
  • Resilience — a truncated/failed call degrades to a fallback row (and verify backstops a failed decide) instead of aborting the run.
  • Row-scoped context — each prompt carries only that claim's images, history row, and relevant requirements, keeping input tokens and cost low.

7. How to run

set -a && . ./.env && set +a            # load GEMINI_API_KEY

# Final predictions (chosen strategy)
python3 code/main.py --pipeline direct --strategy flash_lite \
    --input dataset/claims.csv --output dataset/output.csv

# Batched alternative (fewest requests)
python3 code/main.py --pipeline batched --output dataset/output.csv
